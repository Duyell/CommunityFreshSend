# 社区生鲜配送系统 · 开发任务书（详细目标）

## 一、项目目标

开发一套面向**社区居民 + 生鲜仓库 + 配送人员 + 自提点**的轻量级生鲜电商后端系统，实现**线上选购、下单支付、库存管控、订单履约、多模式配送（配送到家/到店自提/团长自提点）、售后评价**的完整业务闭环；保证系统**高可用、数据一致、接口规范、便于扩展**，支撑社区日常生鲜采购高频访问场景。

---

## 二、业务目标（要实现什么功能）

### 1）用户端业务目标

- 用户注册与登录：
  - 注册页选择身份：**我是居民（买菜）** → 注册即用，角色 `ROLE_USER`
  - 注册页选择身份：**我是配送员（接单）** → 注册后提交配送员申请，管理员审核通过后追加 `ROLE_DELIVERY`，方可接单
  - 商家账号不走注册，由管理员后台直接创建
  - 手机号登录、JWT 鉴权、个人信息维护
- 收货地址管理：新增/编辑/删除/设默认地址
- 自提点选择：选择附近自提点/团长自提点作为提货地址
- 商品浏览：分类展示、新品/热销推荐、商品详情（规格/库存/图片）
- 商品搜索：按名称、分类模糊搜索
- 购物车：添加、修改数量、删除、Redis 缓存（一个用户一个购物车，key: `cart:user:{userId}`）
- 订单管理：
  - 下单（确认商品、选择配送方式、选择送达时段、使用优惠券）
  - 模拟支付：弹出支付窗口，展示随机生成的付款二维码，下方提供"已支付"/"取消支付"两个按钮，支持 15 分钟超时控制
  - 订单列表（按状态筛选）、订单详情、物流追踪
  - 配送中/分拣前可取消订单，申请退款
- 评价管理：收货后商品评分（新鲜度/描述相符）、文字评价、上传图片
- 收藏夹：收藏常购商品，快速加购
- 站内消息：用户登录/刷新页面时，拉取未读消息并弹窗展示（到货提醒、退款进度等）
- 团长申请：申请成为团长、查看申请状态、审核通过后切换团长模式

### 2）仓库（商家）端业务目标

**商家账号说明**：商家不走注册页，由管理员后台直接创建。创建时分配 `ROLE_MERCHANT` 角色，一个平台一个商家。

**1. 商品管理：**

```
分类管理：
  ├── 商品分类树（例：水果 → 热带水果 / 浆果类；蔬菜 → 叶菜 / 根茎；肉禽蛋奶）
  ├── 分类 CRUD + 启用/停用
  └── 前端按分类展示商品

商品管理：
  ├── 新增商品：名称、描述、图片（OSS 上传）、所属分类
  ├── 规格配置：
  │     ├── 固定规格：盒装草莓 300g/盒，售价 ¥19.9，库存按"盒"计
  │     └── 称重计价：散装苹果 ¥8/斤，库存按"斤"计（入库 200 斤，每斤约 3 个）
  ├── 商品状态：上架 / 下架 / 售罄
  └── 编辑商品信息、调整价格
```

**2. 批次库存管理：**

```
批次入库：
  ├── 新建批次：选择商品 → 填写进货批次号、进价、数量、生产日期、过期日
  ├── 同一种商品可有多个批次（不同时间进的货，过期日不同）
  └── 入库后更新该商品总库存

库存扣减 FIFO：
  ├── 订单分拣时，系统自动按过期日升序分配批次（早过期的先出）
  └── 拣货单展示批次信息，指导仓库人员从哪个批次取货

临期与缺货预警：
  ├── 定时任务每日扫描：距过期 ≤ 3 天的批次 → 自动标记"临期" → 前端展示"临期特惠"标签并打折
  └── 商品库存低于安全阈值 → 写入预警记录 → 商家端首页红色提醒"XX 商品库存不足，请及时补货"
```

**3. 订单处理（商家视角全流程）：**

```
待接单 ── 商家查看新订单列表 → 点击"接单"
  │       （可批量接单）
  │
  └── 待分拣 ── 商家/仓库人员按订单逐项拣货
          │
          ├── 固定规格商品：直接取货，数量与订单一致
          ├── 称重商品：称出实重，填入系统（例：下单预估 2 斤，实重 2.3 斤）
          │               └── 系统自动计算差额：多退少补，生成差价调整记录
          │
          ├── 全都有货 → 点击"分拣完成" → 订单进入"待配送"
          │
          └── 部分缺货 → 标记缺货商品行 → 系统自动计算部分退款金额
                          → 退款回到用户账户 → 订单继续进入"待配送"

分配配送员：
  ├── 分拣完成后，商家在订单详情页选择配送员（指派模式）
  └── 或配送员在接单大厅自己抢单（抢单模式），商家无需操作
```

**4. 数据统计：**

```
商家端首页数据看板：
  ├── 今日概况：新订单数、已接单数、待分拣数、待配送数
  ├── 销量统计：按日/周/月统计各商品销量、销售额
  ├── 热销排行：销量 TOP 10 商品
  ├── 客单价分析：平均每单金额
  └── 损耗统计：临期过期报损数量、金额
```

### 3）配送端业务目标

**配送员申请与模式切换：**

```
普通用户（ROLE_USER）进入 App
  │
  ├── 首页 [申请成为配送员] 按钮
  │     │
  │     └── 填写表单 → 提交申请
  │           ├── 姓名
  │           ├── 联系电话
  │           └── 附言（可选）
  │
  ├── 申请提交后 → 按钮变为 [审核中]（置灰不可点）
  │
  ├── 管理员后台审核
  │     ├── 通过 → INSERT user_role（user_id, ROLE_DELIVERY）
  │     │         申请状态变为"已通过"
  │     │
  │     └── 拒绝 → 填写拒绝原因，申请状态变为"已拒绝"
  │
  ├── 审核通过后 → 用户首页按钮变为 [配送员模式]
  │
  └── 点击 [配送员模式] → 进入配送员工作台
        ├── 接单大厅（所有"待配送"状态的订单列表）
        │     每个订单展示：送达地址/自提点、配送时段、商品概要、配送费
        ├── 自由抢单（抢单模式为主，管理员也可后台手动指派）
        ├── 已接订单：取货确认 → 配送中 → 送达确认
        ├── 今日完成数/在途数、历史订单列表、完成率统计
        └── [返回普通模式] → 回到用户首页，继续买菜
```

**配送履约流程：**

```
接单 → 去仓库取货 → 点击"取货确认" → 订单状态变为"配送中"
  │
  ├── 配送到家：逐户送达 → 点击"送达确认" → 订单状态变为"已签收"
  │
  └── 送达自提点：批量交付给自提点/团长 → 点击"送达确认"
        └── 订单状态变为"已送达自提点"
            配送员任务至此完成，后续用户自提由团长负责
```

**配送员权限边界：**

| 可以 | 不可以 |
|------|--------|
| 查看接单大厅所有待配送订单 | 修改订单内容 |
| 自由抢单 | 查看其他配送员的配送记录 |
| 取消已接的配送（取货确认前） | 取货确认后取消配送 |
| 查看自己的配送记录和完成率 | 主动退款（需走平台流程） |

**关键规则：**

| 问题 | 规则 |
|------|------|
| 配送员如何接单 | 接单大厅自由抢单为主，管理员可后台手动指派 |
| 一次能接几单 | 不限，配送员自行判断一趟能带多少 |
| 自提点订单如何算完成 | 送达自提点即算"已送达"，配送员任务完成，后续由团长核销 |
| 配送员取消接单 | 取货确认前可取消，订单回退到"待配送"状态 |

**配送员本质**：配送员 = 普通用户 + `ROLE_DELIVERY` 角色，同一个账号，只是多了一个身份切换入口。

### 4）团长申请与自提业务目标

**团长申请流程：**

```
普通用户（ROLE_USER）进入 App
  │
  ├── 首页 [申请成为团长] 按钮
  │     │
  │     └── 填写表单 → 提交申请
  │           ├── 自提点地址（小区名+门牌号）
  │           ├── 联系人姓名
  │           ├── 联系电话
  │           └── 附言（可选）
  │
  ├── 申请提交后 → 按钮变为 [审核中]（置灰不可点）
  │
  ├── 管理员后台审核
  │     ├── 通过 → 自动执行：
  │     │   1. INSERT pickup_point（owner_type=1, owner_id=当前用户ID）
  │     │   2. INSERT user_role（user_id, ROLE_GROUP_LEADER）
  │     │   3. 更新申请状态为"已通过"
  │     │
  │     └── 拒绝 → 填写拒绝原因，申请状态变为"已拒绝"
  │
  ├── 审核通过后 → 用户首页按钮变为 [团长模式]
  │
  └── 点击 [团长模式] → 进入团长工作台
        ├── 今日待取货列表（选了该自提点的订单）
        ├── 输入提货码核销（扫用户提货码 → 订单状态变为"用户已自提"）
        ├── 历史订单
        ├── 收益统计（佣金，Phase 3 实现）
        └── [返回普通模式] → 回到用户首页，继续买菜
```

**团长权限边界：**

| 可以 | 不可以 |
|------|--------|
| 查看自己自提点的待取货订单 | 查看其他自提点的订单 |
| 输入提货码核销确认取货 | 修改订单内容 |
| 查看历史核销记录 | 主动退款（需走平台流程） |

**团长本质**：团长 = 普通用户 + `ROLE_GROUP_LEADER` 角色，同一个账号，只是多了一个身份切换入口。

**自提业务功能：**

- 自提点管理：平台自营自提点 + 团长自提点信息维护、营业时间、联系方式
- 到货确认：配送员将货物送达自提点后，订单状态变更为"已送达自提点"，同时生成一条站内消息（用户登录/刷新页面时弹窗展示）
- 自提核销：用户出示提货码（6 位数字串，前端可据此生成二维码展示），团长/自提点工作人员在系统中输入提货码完成核销
- 超时未取：超过约定时间 N 天未取，自动退回仓库，同时生成站内消息

### 5）营销与定价

- **优惠券系统**（Phase 1 手动发放，不做领券中心）：
  - 券类型：满减券（满 50 减 5）、折扣券（8 折）、新人券、品类券（仅限水果/蔬菜）
  - 涉及表：`coupon`（券模板）、`user_coupon`（用户持有券）、订单关联优惠券
  - 管理员后台手动给用户发券 → 用户下单时选择可用优惠券 → 订单金额自动扣减
- **起送价 / 配送费 / 包装费**（`sys_config` 配置表维护）：
  - 下单校验：订单金额 < 起送价 → 拒绝下单，提示还差 XX 元
  - 配送费计算：满额免配送费，未满则基础费 + 距离阶梯加价
  - 包装费按件收取
- **配送时段管理**（`order` 表 `delivery_time_slot` 字段，无额外调度逻辑）：
  - 用户下单时选择：上午 (9-12) / 下午 (14-17) / 晚间 (17-20)
  - 配送员接单页面展示时段，自行安排配送顺序
- **临期折扣**（依赖 Phase 2 批次表）：
  - 定时任务每日扫描 `product_batch`，距过期 ≤ 3 天的商品自动打折
  - 前端展示"临期特惠"标签，商品详情页展示过期日期

### 6）系统管理目标

- 权限管理：管理员/商家/配送员/普通用户，RBAC 角色菜单权限控制
- 日志管理：操作日志、订单状态变更日志、异常日志记录
- 系统配置：配送范围、起送价、配送费阶梯、营业时间、自提点管理
- 对账管理：支付流水与订单每日对账

## 三、订单状态流转（核心）

### 配送订单状态机

```
待付款 ──[超时15分钟]──→ 已取消（系统自动取消，库存回补）
  │
  └──[支付成功]──→ 待接单 ──[商家接单]──→ 待分拣
                                            │
                    ┌───────────────────────┤
                    │                       │
                    ↓                       ↓
              分拣完成（全部有货）      部分缺货
                    │                       │
                    ↓                       ↓
               待配送                   部分退款 → 分拣完成
                    │
                    └──[分配配送员]──→ 配送中 ──[送达确认]──→ 已签收
                                                                  │
                                                                  ↓
                                                              待评价 ──→ 已完成
                                                                  │
                                                                  └──[退款申请]──→ 退款审核 → 已退款
```

### 自提订单状态机

```
待付款 ──[超时]──→ 已取消
  │
  └──[支付成功]──→ 待接单 → 待分拣 → 分拣完成 → 待配送 ──[配送员批量送达]──→ 已送达自提点
                                                                              │
                                                                    ┌─────────┤
                                                                    ↓         ↓
                                                               用户已自提   超时未取→退回
                                                                    │       （团长输入提货码核销）
                                                                    ↓
                                                                待评价→已完成
```

**两种配送模式对比：**

| | 配送到家 | 送达自提点 |
|------|---------|-----------|
| 执行人 | 配送员 | 配送员（同一个角色） |
| 配送方式 | 逐户送达 | 批量配送（一趟拉该自提点所有订单） |
| 订单终点 | 用户家门口 | 小区自提点 |
| 用户取货 | 当面签收 | 自行到自提点取货，团长输入提货码核销 |
| 配送效率 | 低（一单一跑） | 高（一趟送十几单） |

关键规则：
- **下单即扣减 MySQL 库存**，超时未付回补库存
- 配送前（待接单/待分拣状态）用户可主动取消
- 配送中状态用户不可取消，需走退款流程
- 所有状态变更记录操作日志（谁、什么时间、从什么状态到什么状态、原因）

## 四、技术目标（要达到什么技术水平）

### 1. 架构规范
- 基于 **SpringBoot + MyBatis-Plus** 分层开发（Controller → Service → Mapper）
- RESTful 接口风格，统一返回结果 `Result<T>`、全局异常处理、参数校验（Bean Validation）
- 接口防重复提交（幂等性）、接口限流

### 2. 数据库设计
- MySQL 主库，合理设计表结构（用户、地址、商品、分类、商品批次、购物车、订单、订单项、支付流水、评价、配送员、自提点、站内消息、优惠券、系统配置）
- 遵循阿里巴巴开发规范：**不使用外键及级联**，数据完整性由应用层保证
- 保证**主键、唯一索引、普通索引、复合索引**正确性
- 核心交易流程必须事务控制

### 3. Redis 深度应用
| 场景 | 方案 |
|------|------|
| 缓存 | 商品列表、分类树、热销数据、用户地址 |
| 购物车 | 用户购物车数据存 Redis（Hash 结构） |
| 分布式锁 | 下单扣库存、支付回调防止重复处理 |
| 购物车 & 库存缓存 | 购物车数据存 Hash；库存缓存读多写少，设置 TTL，定时从 MySQL 全量同步 |
| 缓存问题 | 缓存空值防穿透、互斥锁防击穿、随机 TTL 防雪崩 |

### 4. 消息队列（RocketMQ / RabbitMQ）
- **订单超时取消**：下单后发送延迟消息，15 分钟后检查支付状态，未支付则自动取消 + 库存回补
- **库存同步**：库存变更后异步刷新 Redis 缓存

### 5. 订单核心能力
- **下单原子性**：创建订单 + 扣减 MySQL 库存 + 写入订单项 + 清空购物车 在同一事务中完成
- **模拟支付流程**：
  ```
  用户点击"去支付" → 弹出支付窗口
    ├── 上方：随机生成的付款二维码（纯展示，模拟真实支付场景）
    ├── 下方：[已支付] 按钮 / [取消支付] 按钮
    └── 15 分钟倒计时，超时自动关闭窗口 → 订单取消 + 库存回补

  "已支付" → 订单状态变更为"待接单"，后续正常流转
  "取消支付" → 订单取消，库存回补
  ```
  后端 `PaymentService` 使用策略模式，当前为 `MockPaymentService`，后续可替换为真实支付实现，接口签名不变。
- **称重商品计价**：下单时按"预估重量 × 单价"计算预支付金额，分拣后按实重调整，多退少补
- **部分退款**：缺货商品行级退款，不影响整单其他商品
- **支付幂等**：支付回调通过唯一流水号保证幂等处理

### 6. 安全目标
- JWT 登录鉴权 + Spring Security 接口拦截
- BCrypt 密码加密，不存明文
- 敏感接口防重复提交（Token 机制 + Redis）
- RBAC 权限校验，接口级 + 数据级权限隔离

### 7. 文件存储（阿里云 OSS）

```
上传流程：
  前端选图 → 调后端 POST /api/file/upload → 后端用 AccessKey 上传到 OSS 私有桶
  → 文件以哈希名存储（如 products/a3f9k2.jpg）
  → 数据库不存完整 URL，只存 image_key = "products/a3f9k2.jpg"

访问流程：
  前端需要展示图片 → 调后端 GET /api/file/url?key=products/a3f9k2.jpg
  → 后端用 AccessKey 生成临时签名 URL（有效期 30 分钟）
  → 前端拿到签名 URL 直接加载图片
  → 签名过期后重新请求即可，无需重新上传
```

- 桶设为**私有读写**，不暴露公开访问
- 签名 URL 按需生成，无需存数据库，安全可控

### 8. 性能目标
- 高频读接口 Redis 缓存，降低 DB 压力
- 分页查询、限制大数据量返回
- 支撑数百用户同时访问（社区规模）

## 五、关键业务规则（生鲜场景特有）

### 称重商品 vs 固定规格
| 类型 | 示例 | 计价方式 | 金额确定时机 |
|------|------|---------|------------|
| 固定规格 | 盒装草莓 300g/盒 | 按份定价 | 下单即确定 |
| 称重计价 | 散装苹果 ¥8/斤 | 预估重量 × 单价 | 分拣后按实重多退少补 |

### 库存扣减时机（MySQL 为唯一真相源，Redis 为缓存加速）

```
加购物车 → 不扣库存（仅校验缓存库存 > 0，缓存 miss 穿透查 MySQL）

提交订单  ──MySQL（事务）──→  UPDATE product SET stock = stock - N（真实扣减）
           ──Redis─────────→  DECR product:stock:{id}  N（同步缓存）

支付成功  → 无需操作库存（下单时已扣），只更新订单状态为"已支付"

超时未付  ──MySQL（事务）──→  UPDATE product SET stock = stock + N（回补库存）
           ──Redis─────────→  INCR product:stock:{id}  N（同步缓存）
```

**一致性兜底**：定时任务每 5 分钟从 MySQL 拉取全量商品库存，批量 `SET` 到 Redis。即使中间 Redis 更新失败，5 分钟内自动修复。Redis 缓存设合理 TTL（10 分钟），miss 时穿透查 MySQL。

### 批次管理与 FIFO
- 同一商品可关联多个批次
- 出库时按过期日升序自动分配（早过期的先出）
- 拣货单上展示批次信息，指导仓库人员操作
